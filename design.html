1. 한 줄 컨셉 · 정보 구조 슬로건

1.1 제품 한 줄

시간이 주인공인 장애 대응 도구
대시보드를 줄인 것이 아니라, 경과·신선도·Silence를 시그니처로 둔다.

사용자는 침대에서 한 손으로 조작하는 상황실 OP와, 새벽에 호출당한 개발/운영 담당자다.
3분 안에 “지금 뭐가 터졌고, 어디고, 로그/그래프는, Ack 또는 누구에게 연락할지”만 남긴다.

1.2 이슈 상세 정보 구조 슬로건 (변형 C · 확정)

식별 → 즉시 손 → 필요할 때 깊이

Identify → Act now → Go deep only when needed

이 한 줄이 이슈 상세(및 조치 영역)를 왜 이렇게 쪼개는지의 전부다.
시각 토큰·타이포·상태 규칙이 “어떻게 그릴지”라면, 이 슬로건은 무엇을 먼저 손에 닿게 할지다.

단계 의미 화면에 대응하는 블록
① 식별 지금 이 알림이 무엇인지 · 어디인지 · 얼마나 지났는지 [Main] 이벤트 발생 내용 (기존 이슈 상세 핵심)
② 즉시 손 지금 밟을 초동 행동 [Sub 1] 긴급조치 매뉴얼 (Runbook) — 기본 랜딩
③ 필요할 때 깊이 판단이 막힐 때만 여는 보조 층 [Sub 2] ~ [Sub 4] — 중요도 순 2차 내비

채택 구조명: 변형 C (하이브리드)
Main을 상세 상단에 항상 두고, 그 아래 Sub 영역을 중요도 순 2차 내비 + 한 패널로 둔다.
롱스크롤 전부 펼침·Sub 4탭 동등 취급은 비채택 (§1.5).

1.3 왜 이렇게 구성하는가 (디자인 결정 이유)

문제: 재료가 많아지면 초동이 늦어진다

콘솔 /events 행 확장에는 이미 풍부한 재료가 있다 — 매뉴얼, AI 분석, 유사 이력, 발생 패턴.
모바일에 그대로 나열하면 새벽 한 손 사용자는 다음을 겪는다.

  1. 식별 전에 추정·이력을 읽음 → 사실과 추정이 섞여 오판
  2. 매뉴얼이 화면 아래 묻힘 → “즉시 손”이 스크롤 끝에 있음
  3. 탭 4개가 동등 → 중요도(매뉴얼 > AI > 유사 > 이력분석)가 시각적으로 사라짐
  4. 페이지 왕복 (상세 ↔ 가이드) → 3분 KPI와 충돌

해결: 변형 C가 하는 일

설계 선택 이유 (사용자가 얻는 것)
Main을 접지 않고 상단 성격 유지 사실(발생 내용)을 먼저 고정. AI·이력이 Main을 가리거나 대체하지 않음
Sub1 매뉴얼 = 기본 선택·기본 펼침 식별 직후 손이 가는 곳이 “사람이 발행한 초동 절차”
Sub2~4 = 같은 자리, 칩으로 전환 페이지 전환 없이 깊이만 바꿈. 인지 모델 = “한 이슈의 서랍”
칩 순서 고정 1→2→3→4 중요도가 좌→우 위치로 읽힘 (학습 비용↓)
한 번에 Sub 패널 하나 매뉴얼 체크 중 AI 문단이 끼어들지 않음
로그 전문·그래프 확대만 스택 push “깊이” 중에서도 재료 원문만 별 화면 — Main은 미리보기 유지
근거 없는 AI 비렌더 깊이 층이 추측으로 초동을 오염시키지 않음
하단 Ack | ☎ 유지 어느 Sub에 있어도 상황인지·연락은 항상 닿음

한 문장 정당화

우리는 관제 데이터를 더 보여 주기 위해 화면을 짜지 않는다.
식별을 끝낸 손이 3분 안에 매뉴얼에 닿게 하기 위해 짠다.
AI와 이력은 그 손을 막을 권리가 없고, 막혔을 때만 같은 자리에서 깊이를 연다.

이것이 「식별 → 즉시 손 → 필요할 때 깊이」를 시안·컴포넌트 트리·라우팅에 강제하는 이유다.
토큰 색·폰트 규칙은 이 문장을 배신하지 않는 범위에서만 의미를 갖는다.

1.4 Main / Sub 블록 정의 (변형 C)

[Main] 이벤트 발생 내용 — 식별

기존 이슈 상세의 핵심. 사실 레이어.

  • 히어로: 심각도 · sourceType · 경과 타이머 · 제목
  • 메타: 발생시각 · 시스템/업무(부서) · 호스트 · 오류코드 · 건수 · 담당
  • 재료 미리보기: 로그 상위 1~2행 또는 스냅샷 축소
  • 깊이 출구: 로그 전체 › / (필요 시) 메트릭 확대 — 스택 push
  • 선택: AI 한 줄 티저만 Main 하단 가능 → 탭 시 Sub2 (본문은 Sub2)

[Sub 1] 긴급조치 매뉴얼 (Runbook) — 즉시 손 · 중요도 최고

  • caution → 확인 후 축약
  • 단계 CHECK / ACTION / CONTACT / NOTE · 진행 n/N
  • 명령 [복사] · ACTION은 “콘솔에서 수행함” · CONTACT → ☎ 레이어
  • 진입 시 기본 랜딩 패널
  • 원천: runbook + runbook_step + runbook_rule_mapping

[Sub 2] AI 분석 & 조치제안 — 깊이 1

  • event_ai_analysis kind=EVENT
  • 추정 원인 · 확인 필요 · 권고 조치 · 근거
  • surface-2로 저작 주체 분리 (사람 매뉴얼 ≠ AI 추정)
  • 근거 없는 항목 비렌더

[Sub 3] 유사 이슈 조치내역 — 깊이 2

  • incident_knowledge / similar API
  • 유사도 · status · MTTR · 원인/조치 · [이 조치 재사용]
  • CURATED/VERIFIED 우선 · RAW 하향 · 미상 재사용 금지

[Sub 4] AI 발생이력 분석 — 깊이 3 · 초동 기준 중요도 최저

  • event_ai_analysis kind=HISTORY
  • 반복 패턴 · 시간대 집중 · 과거 해소 요약
  • 기본으로 초동을 점유하지 않음 — 마지막 칩, 필요 시만

2차 내비 (중요도 = 위치)

[ ① 매뉴얼 ]  [ ② AI ]  [ ③ 유사 ]  [ ④ 이력분석 ]
   ▲ 기본 선택
규칙 내용
순서 항상 1→2→3→4. 사용자 설정으로 재배열 금지
기본 선택 Sub1. Sub1 없을 때만 있는 첫 Sub로 폴백
동시 펼침 금지 — 패널 슬롯 하나
배지 Sub1 n/N · Sub3 건수 · Sub2/4 “있음/준비중”
전부 없음 2차 내비·Sub 영역 숨김. Main만 (빈 가이드 금지와 동일)

콘솔 데이터 정합

모바일 블록 콘솔 /events 원천
Main 이벤트 본문·메타 event_history 등
Sub1 긴급조치 매뉴얼 긴급/진급조치 매뉴얼 runbook · runbook_step · by-rule API
Sub2 AI 분석 & 조치제안 AI 분석 (당건) event_ai_analysis EVENT
Sub3 유사 이슈 조치내역 유사이슈 조치이력 incident_knowledge · similar API
Sub4 AI 발생이력 분석 AI 분석 (이력/패턴) event_ai_analysis HISTORY

시안 샘플(실 DB): [AUT] 승인 5xx · Runbook #18 · knowledge CURATED/RAW · qwen2.5.

1.5 비채택 안 (기록)

안 왜 버리는가
A. Main+Sub1~4 롱스크롤 전부 펼침 Sub1이 아래로 밀림 → “즉시 손” 실패
B. Sub 4탭 동등 중요도 소실 · 한 손 타깃 과밀
상세 / 가이드 강제 2페이지 왕복 식별↔손 사이 문맥 단절 · 3분 동선 악화
AI를 Main 히어로 위 추정 > 사실 → 오판·불신

변형 C는 A의 “한 흐름”과 B의 “점프”만 취하고, 각각이 깨는 초동을 버린 타협이다.

2. 디자인 원칙

# 원칙 실천
0 식별 → 즉시 손 → 필요할 때 깊이 Main 사실 고정 · Sub1 기본 랜딩 · Sub24는 칩 전환 깊이 (§1.21.4)

8 읽기 전용 기본 명령 실행 UI 없음. 쓰기 = Ack · Runbook 체크(V1)뿐
9 콘솔과 팔레트 정합 slate 스케일·도메인 의미는 sentinos-console 선례
10 모션 절제 시트 전환 + 타이머 틱만. prefers-reduced-motion 존중
11 사실 ≥ 추정 AI·이력은 Main·Sub1을 대체·선행·가리지 않음. 근거 없으면 비렌더

3.2 색 사용 규칙

  • 심각도: 카드 좌측 스트라이프 3.5px · 뱃지 배경 위주.
    다크 배경 위 소형 텍스트에 심각도 색 단독 사용 금지 (대비 부족).
  • 선택 칩·포커스 카드 외곽선: 액션 블루 (심각도 색과 경쟁 금지).
    Sub 2차 내비의 선택 칩에도 동일 — 심각도 색으로 “선택”을 표현하지 않는다.
  • System Health “GOOD”: 초록 아웃라인 필 등 정상색 계열.

3.3 타이포그래피

용도 폰트 비고
UI / 본문 Pretendard 로컬 자가호스팅, 폴백 시스템 산세리프
데이터 JetBrains Mono + tabular-nums 시간·경과·오류코드·호스트·로그·수치 전부

모노스페이스는 “관제의 재료”를 구분하는 시각 언어다.
본문 산세리프와 섞어 쓰면 안 된다 — 재료는 mono, 설명은 Pretendard.

3.4 시그니처 — 경과 타이머

위치 스펙 동작
인박스 카드 우상단 14px / 700 mono “MM:SS 경과” (활성 이슈만 — Silence 뱃지 없음, v2.1)
이슈 상세 히어로 (Main) 34px / 800 mono 실시간 틱 — 식별 단계의 시간 앵커
상세 · Ack 직후 (같은 화면 체류) 동일 히어로 Silence 만료까지 카운트다운으로 전환
Sub 패널 상단 (선택) 소형 mono Main 타이머와 중복 시 Main만 유지 권장

모든 시간 표시는 서버 시각 오프셋 보정 전제 (R11②).

4. 레이아웃·반응형

4.1 넓이로 가른다 (기기/UA 아님)

뷰포트 모드
~767px 모바일 단일 스택 — 목록 → 상세(Main+Sub) → 로그 전문이 페이지 전환
768px+ (md) 태블릿 마스터·디테일 — 좌 인박스 320px 고정 + 우 디테일(Main+Sub)
  • 회전·리사이즈 시 두 모드 자연 전환 (같은 트리, CSS 분기).
  • 태블릿 우측 = Main+Sub 동일 정보 구조 + 로그 전문 시 스택 push.
  • 태블릿 하단 탭 바 = 좌측 마스터 패널 하단. 우측은 자체 액션 바.

4.2 가로 치수 규칙

  • 기본: % · fr · flex · minmax() · clamp().
  • 고정 px은 규격인 곳만 — 예: 인박스 패널 320px, 터치 타깃 44px. 사용 시 이유 한 줄.
  • 세로는 고정값 허용.

4.3 내비게이션 모드 (폴더 = 답)

모드 UI 하단 해당 화면
탭 내비게이션 탭 바 (이슈 · 이력 · 설정) 탭 루트
스택 내비게이션 액션 바 (Ack | ☎ 관계자) 상세(Main+Sub) · 로그 상세

탭 바와 액션 바는 같은 자리를 두고 배타 경쟁.
usePathname() 분기로 숨기지 말고, 라우트 그룹(tabs / stack)으로 표현한다.

변형 C 보충: Sub 전환은 스택 push가 아니라 상세 내부 2차 내비다.
“즉시 손 ↔ 깊이”를 페이지 경계로 나누지 않는다.

4.4 WebView · 모바일 크롬

  • safe-area: 하단 탭/액션/시트에 env(safe-area-inset-bottom).
  • 높이: 100dvh ( 100vh 금지 ).
  • 문서 오버스크롤 바운스 억제. 시트·Sub 패널 내부 스크롤 격리.
  • -webkit-tap-highlight-color: transparent.
  • 입력 font-size >= 16px (iOS 줌 방지).
  • viewport width=device-width (300ms 탭 지연 제거).
  • 관계자: 네이티브 tel: / sms:.

5. 상태 6종 — 모든 화면 필수

미구현 상태 화면은 미완성이다.

상태 시각·카피 방향
빈 인박스: “지금은 조용합니다 · 마지막 확인 HH:MM”
로딩 스켈레톤 (레이아웃 점프 최소화)
오류 원인 + 다음 행동 — “로그를 불러오지 못했습니다 · 다시 시도”
권한 없음 블러 + 자물쇠 오버레이 / 액션 🔒
오프라인 네트워크 단절 명시 + 재시도
스테일 상단 배너 “HH:MM 이후 갱신 실패 — 표시 중인 목록은 과거입니다” + 목록 뮤트

빈(200 조용함) 과 5xx/스테일(고장) 을 같은 빈 상태로 합치지 않는다.

Sub 영역 상태: 매뉴얼 없음 · AI 준비 중 · 유사 0건은 각자 패널 카피로 처리하고, Main 식별은 유지한다.

6. 정보 위계 — 화면별 컨셉

화면 1 — 이슈 인박스 (/issues)

목표: 0.5초 안에 “몇 건 · 가장 급한 것 · 얼마나 지났는지”.
(목록 단계의 식별 입구 — 상세 슬로건의 전 단계.)

  • 헤더: “이슈” + System Health 뱃지(웹 산정값) + mono 캡션 “마지막 확인 · 1분마다 갱신”.
  • 필터: 전체 / Critical / Major / Minor (액션 블루 선택).
  • 카드: 1행 — 심각도 라벨(mono) + 경과 타이머
    2행 — 오류 요약 (강조)
    3행 — mono 메타 (시스템·업무·부서 · 담당 1인 · 발생 시각)
  • 좌측 심각도 스트라이프.
  • Ack된 카드는 목록에 없음 (v2.1). Silence 뱃지 디자인은 인박스에 적용하지 않음.
  • 새 이슈 배너 · 스크롤 점프 금지.

화면 2 — 이슈 상세 (/issues/[id]) · 변형 C 본진

목표: 「식별 → 즉시 손 → 필요할 때 깊이」를 한 화면에서 완수.

┌ 앱바 ‹ · 호스트 · (+새이슈)
├ [Main] 히어로 + 메타 + 재료 미리보기     ← ① 식별
├ [2차 내비] ①매뉴얼 ②AI ③유사 ④이력분석  ← 중요도 순
├ [Sub 패널] 기본 = Sub1 매뉴얼            ← ② 즉시 손 / ③ 깊이
└ 액션 바  Ack | ☎

① 식별 — Main

  1. 앱바
  2. 히어로 — 심각도·sourceType · 건수 · 대형 경과 · 제목 · mono 메타 · 담당 목록
  3. sourceType 미리보기 — 로그 1~2행 또는 스냅샷 축소 + 전체 ›
  4. (선택) AI 한 줄 티저 → Sub2

히어로는 surface, 스트라이프 4px. Main은 사실 전용 — 추정 본문을 넣지 않는다.

② 즉시 손 — Sub1 (기본 랜딩)

  • caution → Runbook 단계 체크리스트
  • 진행 mono n/N · 체크 1탭 / 해제 확인
  • 저작 주체 = 사람 발행 정본

③ 필요할 때 깊이 — Sub2~4 (칩 전환, 동시 펼침 없음)

칩 블록 역할
② AI 분석 & 조치제안 당건 추정·권고 (근거 필수)
③ 유사 이슈 조치내역 재사용 코퍼스
④ AI 발생이력 분석 패턴·2차 맥락

하단 액션 바는 Sub와 무관하게 고정 — 손이 매뉴얼에 있든 AI에 있든 Ack/연락 가능.

화면 3 — 로그 상세

목표: Main에서 연 재료 깊이 — 원문 가독 + 오류 하이라이트 + 시간 문맥.

  • mono 8~12px, 줄바꿈 유지, Critical 틴트 하이라이트.
  • 마스킹 ▮▮▮, 스택 잘림 없음.
  • 전후 로그 · 복사 · 공유(콘솔 링크).

화면 4 — Ack 바텀시트

목표: 필수 3필드 빠르게 · Silence 의미 명확.

  • dim + 고정 높이 시트. 키보드 올라와도 저장 버튼 가림 금지.
  • 조치예정 구간 mono 캡션. 2h 경고. 원인 퀵 칩. 재사용 출처 뱃지.
  • 풀폭 액션 블루 CTA: “저장 — Ack 처리 · 알림 해제”.
  • 토스트: “상황인지 완료 처리됨 · 알림 해제”.

화면 5 — 관계자 연락 레이어

목표: 이탈 없이 전화/SMS.

  • Ack와 슬롯 공유. 행: 이니셜 · 이름/역할 · 전화 mono · 전화 / SMS.
  • 캡션: SMS 요약 자동 첨부.

화면 7 — 이력

목표: 재발 패턴 + 조치 재사용 코퍼스 (인박스 밖 탐색).

  • 검색 · 필터 칩. 그룹 카드에 재발 횟수 mono/700.
  • 원인 미확정 = 뮤트 · 재사용 버튼 없음.
  • 상세 Sub3과 같은 지식 원천, 진입 맥락만 다름 (이슈 안 vs 탭 탐색).

화면 8 — 설정

목표: 신원·권한 확인 · 테마 · 버전. 잡동사니 금지.

  • 카드 3 + 점선 예약(알림 V1).
  • 테마 세그먼트: 다크(기본) | 라이트 | 시스템.

7. 인터랙션·카피

7.1 인터랙션

규칙 내용
터치 타깃 ≥ 44px. Ack 영역과 ☎ 영역 분리
Sub 칩 중요도 순 고정 · 한 패널만 활성
시트 Ack / 관계자 동시 펼침 없음
그래프 팬/줌/탭 없음 (Main 미리보기·확대 동일)
명령어 실행 버튼 없음 · [복사] 만
모션 시트 + 타이머 틱 · reduced-motion

7.2 카피 톤

  • 원인 + 다음 행동 (“불러오지 못했습니다 · 다시 시도”).
  • 성공은 결과 상태 언어 (“상황인지 완료 처리됨 · 알림 해제”).
  • 빈 화면은 안심 + 신선도 (“지금은 조용합니다 · 마지막 확인 HH:MM”).
  • 스테일은 과거 데이터임을 명시.
  • Sub 빈 상태 예: “등록된 긴급조치 매뉴얼이 없습니다 · AI 제안 또는 유사 이력을 확인하세요”.

8. 컴포넌트 재사용 경계

구현 시 공통으로 분리할 단위:

  • 심각도 스트라이프 카드
  • 경과 타이머 (크기 variant)
  • Main 히어로 블록
  • Sub 2차 내비 칩 + 패널 슬롯
  • Runbook 단계 카드 · AI 분석 블록 · 유사 이력 카드
  • 하단 탭 바 / 하단 액션 바
  • 바텀시트 셸 (Ack · 관계자)
  • 스테일 배너 · 토스트 · 필터 칩
  • 권한 블러 오버레이 · 마스킹 텍스트

shadcn/ui 변수 체계와 호환. 토큰은 CSS 변수(@theme / :root)로만 주입 — 컴포넌트에 매직 컬러 하드코딩 금지.

10. 검증 방법

  1. 규칙 단위로 시안·구현 대조 (토큰·위계·상태 6종·가려짐 2종).
  2. 정보 구조 검증: 상세 진입 시 Sub1이 기본인지 · AI가 Main 위에 오지 않는지 · Sub 동시 펼침이 없는지.
  3. ?demo= 픽스처로 상태 6종 재현.
  4. 768px 경계에서 스택 ↔ 스플릿 전환.
  5. warroom-op · dev-operator 페르소나 리뷰 후 반영.
  6. 토큰 값 임의 변경 PR은 거부.
  7. 슬로건 위반 PR은 거부 — “데이터 더 보여 주기”를 이유로 Main/Sub 위계를 무너뜨리면 안 된다.
